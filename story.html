The idea

Where gapNinja came from and what it was actually trying to fix.

The complaint

Job search tools tell you a number, not the truth

Most tools give you a vague "match score" with nothing behind it — you can't act on 72%. What you actually need to know is which specific skills are missing, so you can decide whether to close the gap or apply anyway.

The bet

Track the search, don't try to run it

gapNinja doesn't source jobs or auto-apply. You bring the posting; it scores your fit against it honestly, drafts the letter, and remembers everything — because most people's "system" is a spreadsheet they stop updating by week two.

How it actually got built

Four days, in order — including the decisions that got reversed, because those are the ones worth telling.

Day 1 — Idea & core loop

The matching engine came before anything else

Before any UI polish: resume parsing, a skills taxonomy, and a gap-scoring algorithm. The plan started as a Next.js app, but the dev environment's npm registry was blocked mid-build — so the whole stack pivoted to plain HTML/CSS/JS with no build step, no framework, no dependency to fight. A constraint that turned into the right call.

Day 1–2 — Real infrastructure

From local storage to a real backend, without slowing down

Firebase Auth and Firestore replaced browser localStorage, with per-user security rules written from the start (request.auth.uid == userId) so the architecture wasn't storing everyone's resumes in one shared bucket. An admin role came later, deliberately scoped read-only by default — writable only where a feature genuinely needed it, like replying to a support ticket on someone else's behalf.

Day 2 — Building on feedback, not a spec

Shipped a feature, watched it not land, cut it

A full AI-powered resume rewriting feature got built end-to-end — a Cloud Function calling Claude, client wiring, graceful fallback. The output didn't feel right. Rather than iterate on something with the wrong foundation, the entire feature came out: the UI card, the Cloud Function, every reference — cleanly, with a subagent auditing cross-file dependencies first so nothing broke on the way out.

Day 2 — Small feedback, fast turnaround

"Make them stack vertical" and the layout was fixed same-session

A screenshot of uneven cards in a 3-column grid turned into a single-column stacked layout within the same conversation — small, but it's the loop that matters: real UI feedback, not assumptions, driving every visual decision.

Day 3 — A full support system, end to end

Tickets, sequential numbers, and a race condition solved with a database transaction

A support form seems simple until two people submit at the same instant and both need a unique ticket number. That's solved with a Firestore transaction on a shared counter — atomic read-then-increment, no collisions — plus a search across ticket number, email, and subject on the admin side.

Day 4 — Business layer, backed by research

Pricing wasn't guessed — it was researched, then argued with

Before writing a single price on the page: real 2026 freemium conversion benchmarks, a competitor teardown (Teal, Huntr, Careerflow), and a specific pushback on an initial "cap free users at 10 saved jobs" idea — landed on 15 instead, because every direct competitor was more generous, and the number needed to hold up next to them.

Day 4 — The privacy audit nobody asked for

Before the repo went public, a full history search — and one real find

Every commit, not just the current files, got searched for anything that shouldn't be public. The good news: no API keys or user data had ever been committed. The actual finding was subtler — a personal email baked into every commit's author metadata, invisible unless you knew to look for it. Fixed by rewriting the entire history into a single clean commit under a non-personal identity, with the full original history preserved on a local-only backup branch first, in case anything needed to be recovered.

How this got built, as a set of rules

Not a mission statement — just what actually happened, repeatedly, across four days.

Cut sunk cost fast

The AI resume builder took real effort to build. It came out anyway, cleanly, the moment it wasn't working — no attachment to the hours already spent.

Research before committing

Pricing, freemium caps, and infrastructure cost projections were all grounded in real 2026 data before a single number went on the page.

Security isn't a later problem

Per-user data isolation was in the rules from the first Firestore collection — and privacy got a full audit before the repo ever went public, not after.

The parts of the plan that are designed but deliberately not built yet.
